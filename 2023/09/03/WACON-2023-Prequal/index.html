mosaic

김00씨의 문제다..

요런 구조로 문제파일이 제공되고,

요런 flask app.py가 주어진다.

코드를 적당히 분석하면 아래와 같다.

  1. 회원가입과 로그인 기능(sqli 불가능)
  2. 이미지 파일 업로드 기능
  3. 업로드한 이미지 파일을 모자이크 하는 기능
  4. 업로드된 파일을 확인하는 기능

그리고 도커파일을 보면 flag는 /test-flag.png에 존재하고 코드에서 copyfile(FLAG, f'{UPLOAD_FOLDER}/{session["username"]}/flag.png')를 통해 uploads/admin/flag.png로 복사된다.

그리고 /mosaic 엔드 포인트를 보면 누가봐도 ssrf 백터인 image_data = requests.get(image_url, *headers*={"Cookie":request.headers.get("Cookie")}).content이 존재한다.

문제의 흐름을 보면 일단 어떤 방법으로 admin 세션을 따서(1) ssrf백터를 통해서 127.0.0.1:9999로 요청을 보내면 된다. 이때 125줄의 image_url.startswith("http://") or image_url.startswith("https://"):를 우회해야 한다.

  1. admin session leak
  2. image_url.startswith("http://") or image_url.startswith("https://"): bypass
  3. ssrf
  4. get Flag!!

먼저 '/check_upload/@<username>/<file>' 엔드 포인트를 보면 username이 admin이고 usersession이 admin이 아닐시에만 Access Denied..를 리턴하고 나머지는 send_from_directory(f'{UPLOAD_FOLDER}/{*username*}', *file*) 해주는것을 알 수 있다. 이때 username에 ../를 입력하고 file에 password.txt를 입력하면 path traversal이 발생해 uploads/../password.txt의 내용을 알려준다.

이제 admin session을 땄으니 http://와 https:// 필터링을 우회해야한다.

사실 여기 언인텐으로 HTTP://로 우회하는 방법이 있었지만 인텐은 아래 코드의 url.lstrip()으로 인해 http://로 우회하는 것이였다.

이제 필터링까지 bypass했으니 http://127.0.0.1:9999/#.png로 uploads/admin/flag.png를 만들고 요청 해서 flag를 딸 수 있다.

  1. path traversal(uploads/../password.txt)
  2. /mosaic => http://127.0.0.1:9999/#.png
  3. /check_upload/@admin/flag.png

warmup-revenge

이건 트위터에서 잠깐 봤던게 문제로 나왔다.. 좀 자세히 봐놓을껄..

뭐가 굉장히 많은데 인터페이스랑 코드스타일이 약간 hspace ctf에서 봤던 느낌이었다. bot이 있는걸 보면 xss일텐데 아무리봐도 board쪽에는 다 clean_html이 박혀있어서 백터가 없었다. xss백터는 downlaod.php에 있었는데 코드는 아래와 같다.

header("content-disposition: attachment; filename=\"$original\"");부분에서 취약점이 발생한다.

먼저 content-disposition는 HTTP Response Body에 오는 컨텐츠의 기질/성향을 알려주는 속성이다. 위 코드에서는 attachment로 설정돼있는데 이는 filename과 함께 body에 오는 내용을 다운받으라는 뜻이다. 이걸 무력화하게 되면 body의 내용이 그대로 렌더링되서 XSS를 트리거할 수 있다.

마음까지 찍어놓은 트윗이었는데..

아무튼 %0D를 통해서 헤더를 지워버릴 수가 있고 이로 인해서 xss를 트리거 할 수 있다.

좀 더 근본적인 이유를 찾아 php-src를 보면 아래와 같다.

https://github.com/php/php-src/blob/master/main/SAPI.c#L739

요런 코드에서 a.php?inp=a.html%0D로 요청을 보내면 a.html이 렌더링 되면서 xss가 트리거 된다.

하지만 이렇게 보내보면 csp때문에 아래와 같은 오류가 발생한다.

드림핵에서 비슷한 문제를 겪은적이 있는데 그냥 download에 Payload를 올려두고 script src로 불러오면 해결할 수 있었다.

여기서 burp의 hex를 이용해서 a의 hex인 61을 \r의 hex인 0d로 치환한다.

그리고 script src를 이용해서 /download.php?idx=1008를 불러온다.

이를 통해서 flag를 가져올 수 있다.
